# 消息模式规范

## 目标

消息模式规范用于降低重复投递、乱序消费、无限重试、死信不可追踪和链路上下文丢失的风险。异步处理必须默认假设消息可能重复、延迟或失败，并用可验证的模式保证业务结果一致。该规范为 AsyncAPI 契约之外的运行时可靠性提供统一要求。

## 适用范围

本规范适用于 `apps/`、`services/`、`src/`、`backend/` 中消息生产者、消费者、队列、主题、重试配置、死信处理、补偿任务、批处理消费和集成测试。它也适用于事件驱动前端或移动端订阅消息时的重复处理边界。它不适用于进程内事件总线、单机内存队列或仅用于日志采集且不驱动业务状态的消息流。

## 规则

1. 消息消费者必须按至少一次投递模型设计，能够识别并安全处理重复消息；不得假设 broker 只投递一次。
2. 每条业务消息必须有稳定 `messageId` 或业务幂等键；消费者必须记录已处理标识或使用业务唯一约束防止重复副作用。
3. 生产者发布消息应当与本地状态变更保持一致，可以使用事务外盒、可靠发布或等价机制；不得在状态提交前发布不可回滚事件。
4. 重试策略必须设置最大次数、退避策略和可观测记录；不得无限重试或无间隔高频重试。
5. 不可恢复错误必须进入死信或隔离队列，并保留原始消息、错误码、失败原因和追踪上下文；不得静默丢弃。
6. 消费者必须区分可重试错误、不可重试错误和幂等重复请求；不得对校验失败或契约不兼容消息反复重试。
7. 跨服务消息必须传播追踪上下文，并在消费、重试、死信和补偿路径中记录同一关联标识。
8. 批量消费必须定义单条失败时的提交策略；不得因一条坏消息导致整批永久阻塞。

## 示例

正例：

```python
def handle_order_paid(message):
    key = message["messageId"]
    with transaction():
        if not processed_messages.insert_if_absent(key):
            return
        mark_invoice_paid(message["data"]["orderId"])
```

反例：

```python
while True:
    try:
        consume(message)
        break
    except Exception:
        continue
```

## 检查清单

- 消费者是否能安全处理重复消息，并记录幂等键或使用唯一约束。
- 生产者发布和本地状态变更是否有一致性方案。
- 重试是否有最大次数、退避策略、分类和可观测记录。
- 不可恢复错误是否进入死信或隔离队列，并保留诊断字段。
- 追踪上下文是否贯穿发布、消费、重试和死信路径。
