# 韧性设计规范

## 目标

韧性设计规范用于降低下游故障放大、无限等待、重复副作用、过载雪崩和降级不可验证的风险。服务必须为调用、排队、重试、限流、熔断和降级定义明确边界，并把失败路径暴露为可观测信号。这样调用方、运维和评审者可以判断系统在依赖异常时是否保持受控行为。

## 适用范围

本规范适用于 `apps/`、`services/`、`src/`、`backend/` 中 HTTP API、gRPC 服务、外部依赖客户端、消息消费者、批处理任务、后台作业、公共框架中间件、配置文件、部署参数和集成测试。它覆盖超时、重试、退避、幂等、限流、熔断、隔离、降级、错误映射和故障注入验证。它不适用于无外部依赖、无共享资源、无生产流量影响的本地一次性脚本。

## 规则

1. 所有网络、数据库、缓存、消息和文件存储调用必须声明超时或截止时间；不得使用无限等待或依赖框架隐式默认值。
2. 重试必须限定次数、退避策略、可重试错误类型和幂等前提；不得对非幂等写操作、输入错误、权限错误或契约错误自动重试。
3. 服务入口、昂贵操作和下游依赖调用应当具有限流或并发隔离；不得让单个租户、批任务或依赖故障耗尽全局线程、连接或队列资源。
4. 对关键下游依赖必须定义熔断或快速失败策略，并暴露熔断打开、半开、关闭和拒绝计数；不得在依赖持续失败时继续全量压测下游。
5. 降级路径必须声明用户可见行为、数据新鲜度、功能边界和恢复条件；不得静默返回错误数据或把降级伪装为完整成功。
6. 消息消费和后台任务必须处理重复投递、部分失败、死信或补偿边界；不得通过无限重投扩大故障。
7. 韧性策略必须通过配置或框架默认值集中管理，并允许服务按依赖覆盖；不得把环境差异和魔法常量散落在业务逻辑中。
8. 超时、重试耗尽、限流、熔断、降级和死信必须记录日志、指标或追踪信号；不得让失败控制只存在于代码路径中。
9. 新增或修改韧性策略必须包含测试或演练证据，覆盖依赖超时、错误响应、过载和恢复；不得只验证正常路径。

## 示例

正例：

```yaml
dependencies:
  payment-service:
    timeout: configured-by-service
    retry:
      enabled: true
      retryableErrors: [timeout, transient-unavailable]
      requiresIdempotency: true
    circuitBreaker:
      enabled: true
    fallback: "mark order as payment_pending"
```

反例：

```python
while True:
    response = requests.post(payment_url, json=payload)
    if response.ok:
        break
```

## 检查清单

- 每个外部调用、数据库调用、队列操作和文件存储操作是否有明确超时。
- 重试是否限定错误类型、次数、退避和幂等前提。
- 限流、隔离或队列容量是否能防止单一路径耗尽共享资源。
- 熔断、快速失败和降级是否声明触发、用户可见行为和恢复边界。
- 超时、重试耗尽、限流、熔断、降级和死信是否有日志、指标或追踪信号。
- 测试或演练是否覆盖依赖失败、过载、重复消息和恢复路径。
