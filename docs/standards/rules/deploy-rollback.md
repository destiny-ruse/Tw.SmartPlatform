# 部署与回滚规范

## 目标

部署与回滚规范用于降低发布不可验证、失败后无法恢复、事故信息缺失和运行状态不可追溯的风险。每次部署必须能说明发布了什么、如何确认可用、何时触发回滚以及回滚到哪个制品。规范目标是让发布动作、验证结果和事故记录可被后续审查与改进。

## 适用范围

本规范适用于应用、后台任务、前端静态站点、容器服务、Kubernetes 工作负载、配置变更和数据库兼容发布的部署与回滚说明。它不适用于开发者个人环境、一次性演示环境或不承载真实用户流量的本地调试部署。

## 规则

1. 部署记录必须包含目标环境、制品版本、镜像标签或包版本、配置来源、提交 SHA 和执行时间；不得只记录“已发布”。
2. 部署前必须确认候选制品通过对应 CI 门禁，并存在可用的上一版本或明确的回滚候选；不得发布无法定位回滚目标的制品。
3. 部署步骤必须区分发布、验证和完成记录；不得把健康检查失败的部署标记为成功。
4. 部署后必须执行健康检查、关键路径探测或等效验证，并记录验证结果；不得只依赖进程启动或容器 Running 状态判断业务可用。
5. 回滚触发条件必须与错误率、健康检查、关键业务验证或人工确认的异常现象相关；不得在事故中临时猜测回滚标准。
6. 配置、数据库和消息契约变更必须说明是否向前兼容和向后兼容；不得发布只能通过删除数据或破坏客户端恢复的变更。
7. 回滚执行后必须再次验证服务健康和关键路径，并记录回滚版本、触发原因和残留风险。
8. 发生发布事故时必须补充事故笔记，至少记录影响范围、发现方式、处置动作、当前状态和后续跟进项；不得用聊天记录替代仓库内或工单内的可检索记录。

## 示例

正例：

```yaml
deployment:
  environment: prod
  image: registry.example.com/order-api:1.4.0-abc1234
  commit: abc1234
  verify:
    - /healthz
    - GET /api/v1/orders/ping
  rollback_to: registry.example.com/order-api:1.3.9-def5678
```

反例：

```text
今晚已上线，失败就手动看情况回滚。
```

## 检查清单

- 部署记录是否包含环境、制品、配置、提交和时间。
- 发布前是否确认 CI 门禁通过，并存在明确回滚候选。
- 部署后是否执行健康检查和关键路径验证，结果是否留痕。
- 配置、数据库和契约变更是否具备兼容或恢复说明。
- 回滚触发条件、回滚目标和回滚后验证是否清晰。
- 发布异常是否形成可检索的事故笔记和后续项。
