# 指标规范

## 目标

指标规范用于降低服务健康只能靠日志推断、告警口径不一致、标签基数失控和趋势分析不可复现的风险。服务必须把关键技术和业务信号暴露为命名稳定、标签受控、可聚合的指标。这样容量、SLO、告警和发布评审可以基于同一组可查询信号进行判断。

## 适用范围

本规范适用于 `apps/`、`services/`、`src/`、`backend/` 中 HTTP API、gRPC 服务、消息消费者、批处理任务、后台作业、网关适配器、基础设施组件、指标配置、仪表盘和告警查询。它覆盖指标命名、类型选择、标签、基数、聚合窗口、直方图分桶、业务指标和评审查询准备。它不适用于一次性本地性能实验的临时输出，也不替代财务、审计或数据仓库中的离线统计口径。

## 规则

1. 服务必须暴露请求量、错误量、延迟、依赖调用、队列或任务处理结果等核心指标，并使用稳定命名；不得把可告警信号只写入日志。
2. 指标名称必须表达资源、动作和度量含义，并在单位不明显时包含单位后缀；不得在同一指标中混合毫秒、秒、字节或计数。
3. 标签必须限制为低基数字段，例如 `service`、`environment`、`version`、`operation`、`status`、`dependency`；不得使用用户 ID、订单 ID、traceId、手机号、错误消息或原始 URL 作为标签。
4. 错误、超时、限流、熔断、降级和重试耗尽必须有可聚合计数器或状态指标；不得只依赖人工搜索日志判断影响范围。
5. 延迟指标应当使用直方图、摘要或平台支持的分位聚合方式，并声明查询窗口；不得用单次耗时样本替代服务级趋势。
6. 仪表盘和告警查询必须使用明确聚合窗口、过滤维度和分组维度；不得使用依赖当前 UI 默认时间范围的口头查询。
7. 新增指标必须说明消费场景，例如 SLO、容量、发布验证、故障定位或业务趋势；没有消费场景的高成本指标应当删除或降采样。
8. 指标变更必须保持向后兼容或提供迁移说明；不得静默改名、改单位或改变标签语义导致既有告警失效。

## 示例

正例：

```yaml
metrics:
  - name: http_server_requests_total
    type: counter
    labels: [service, environment, operation, status]
  - name: dependency_request_duration_seconds
    type: histogram
    labels: [service, dependency, operation, outcome]
```

反例：

```json
{
  "metric": "request_latency",
  "labels": {
    "userId": "usr_9f2a1c",
    "url": "/orders/202604270001/items/8842"
  }
}
```

## 检查清单

- 核心请求、错误、延迟、依赖、队列和任务结果是否有可查询指标。
- 指标名称、类型和单位是否稳定且语义单一。
- 标签是否控制基数，并排除用户 ID、订单 ID、traceId 和原始 URL。
- 聚合窗口、过滤维度和分组维度是否在仪表盘或评审说明中明确。
- 错误、超时、限流、熔断、降级和重试耗尽是否有独立计数或状态信号。
- 指标改名、改单位或标签变化是否提供迁移说明并保护现有告警。
