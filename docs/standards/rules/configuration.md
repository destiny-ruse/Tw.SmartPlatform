# 配置管理规范

## 目标

配置管理规范用于降低环境差异、默认值不明、密钥误用和启动后才暴露配置错误的风险。它要求配置来源、覆盖顺序、校验和运行时可观测性保持一致，使后端服务、任务和公共框架能力可以被稳定部署和审查。

## 适用范围

适用于后端服务、批处理任务、消息消费者、公共框架库、部署清单、配置文件、环境变量绑定和启动参数。也适用于 API 客户端、数据库、缓存、消息、观测和安全相关配置的代码评审；不适用于用户可在产品界面自行保存的业务偏好或运行时实验开关的业务规则本身。

## 规则

1. 配置来源优先级必须明确，推荐顺序为代码安全默认值、基础配置文件、环境配置文件、环境变量、启动参数；同名项的覆盖行为必须可预测。
2. 配置键必须按功能域分组并使用稳定名称，不得在业务逻辑中散落字符串 key 或以环境名拼接配置路径。
3. 非密钥配置必须提供安全默认值或显式说明必填；生产环境缺失必填配置时必须启动失败。
4. 密钥、令牌、证书和连接串必须以引用或注入方式获得，不得把明文密钥写入仓库配置、示例日志或默认值。
5. 启动时必须校验配置类型、取值范围、必填项和组合约束，校验失败应当输出不含敏感值的配置路径和失败原因。
6. 环境覆盖必须只表达部署差异，不得改变业务语义、权限边界或数据模型。
7. 配置变更影响超时、重试、缓存、限流或外部依赖时，应当在评审中说明行为变化和回滚方式。
8. 运行时可见的配置摘要可以暴露非敏感键、来源和是否使用默认值，但不得输出密钥值或可反推密钥的片段。

## 示例

正例：

```yaml
catalog:
  cache:
    ttlSeconds: 300
  downstream:
    timeoutMs: 2000
    retryMaxAttempts: 2
    apiKeyRef: ${SECRET_REF_CATALOG_API_KEY}
```

正例说明：普通配置有明确类型和默认边界，密钥通过引用注入。

反例：

```yaml
catalog:
  downstream:
    timeoutMs: 0
    retryMaxAttempts: 999
    apiKey: "prod-plain-text-key"
```

反例说明：无界超时和重试会放大故障，明文密钥不得进入配置文件。

## 检查清单

- 是否说明配置来源优先级以及环境变量、启动参数的覆盖规则。
- 必填项、默认值、类型和取值范围是否在启动时校验。
- 密钥是否只以引用或注入形式出现，日志和错误信息是否不会泄露敏感值。
- 环境配置是否只表达部署差异，没有把业务规则硬编码到环境名。
- 配置变更是否说明对缓存、重试、数据库、消息或外部依赖的影响。
- 是否有回滚配置变更或恢复默认值的明确方式。
