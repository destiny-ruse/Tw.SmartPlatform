# 代码评审规范

## 目标

代码评审规范的目标是降低缺陷、回归、安全问题和不可维护变更进入主干的风险。
评审必须把正确性和可验证性放在个人偏好之前，让提交者和评审者使用一致的严重级别、反馈方式和合并判断。
一致的评审口径可以让问题被明确定位、被直接修复，并在例外发生时留下可追溯记录。

## 适用范围

适用于 pull request、merge request、补丁评审、自动生成代码复核，以及涉及 `src/`、`tests/`、`docs/`、`tools/`、CI 配置、部署配置、数据库迁移和 API 契约的变更。
本规范覆盖评审问题分类、反馈写法、测试缺口识别、合并前确认和例外记录。
它不替代语言级格式化规则、架构决策流程或事故复盘；这些内容应当引用对应规范或流程。

## 规则

1. 评审必须优先检查正确性、数据安全、权限边界、兼容性、可观测性、回滚风险和缺失测试，不得先用命名或排版偏好主导评审结论。
2. 阻断性问题必须标记为必须修复，并说明会导致的失败场景、破坏的契约或不可接受的运行风险。
3. 非阻断性建议应当标记为应当或可以，并说明它对可维护性、可读性或后续演进的影响；不得把个人偏好伪装成质量门禁。
4. 每条评审意见必须定位到具体文件、行号、配置项、接口或测试场景，并给出可执行的修正方向或需要提交者回答的问题。
5. 评审必须识别新增、修改或删除行为对应的测试缺口；当缺少单元、集成、契约、端到端或人工验证时，必须要求补充测试或记录例外。
6. 合并前必须确认阻断性意见已修复或已按治理流程记录例外，相关测试、文档、契约和生成索引已同步更新。
7. 评审者不得要求提交者重写与本次风险无关的大范围代码；必要的重构应当明确与当前变更风险的关系。
8. 自动生成或批量生成的代码必须说明已检查的范围和未覆盖的风险，不得声称通过没有运行或没有证据支持的验证。

## 示例

正例：

```text
必须修复：src/orders/createOrder.ts:84 在库存扣减失败后仍提交订单状态。
这会让支付成功但库存未锁定的订单进入后续流程。请在事务失败时回滚订单状态，
并补充覆盖库存服务返回 409 的集成测试。
```

反例：

```text
这里看起来不太舒服，建议整体重写一下。
```

## 检查清单

- 是否先检查正确性、安全、兼容性、回归风险和测试缺口。
- 是否为阻断性意见说明失败场景、影响范围和修复方向。
- 是否把风格偏好与必须修复的问题区分清楚。
- 是否确认新增或修改行为有对应的自动化测试或明确人工验证记录。
- 是否确认契约、文档、迁移、CI 或生成文件已随代码同步更新。
- 是否记录未解决争议、例外理由和后续跟踪位置。
