# 前端可访问性规范

## 目标

前端页面必须让键盘用户、辅助技术用户和不同输入设备用户都能完成核心流程，避免只对鼠标和视觉用户可用。统一可访问性规则可以把语义、焦点、状态提示和颜色对比纳入日常评审，而不是发布前补救。

## 适用范围

适用于 Web 前端页面、Vue 组件、表单、弹窗、导航、图标按钮、状态反馈、错误提示、路由切换和设计实现验收。纯后端接口、不可交互的内部截图和第三方不可控嵌入内容不属于本规范范围；第三方组件被封装为项目组件时必须补足可访问性约束。

## 规则

1. 可使用原生语义元素时必须优先使用 `button`、`a`、`nav`、`main`、`form`、`label` 等元素，不得用无语义 `div` 模拟交互控件。
2. 所有可交互控件必须支持键盘访问、可见焦点和可预期的 Tab 顺序。
3. 图标按钮、输入框、选择器和自定义控件必须提供可被辅助技术读取的名称，例如文本、`aria-label` 或关联 `label`。
4. 表单错误、异步加载、保存成功和危险操作结果应当以文本状态呈现，不得只依赖颜色、图标或短暂动画。
5. 弹窗、抽屉和菜单必须管理焦点进入、焦点返回和 Escape 或明确关闭方式。
6. 颜色、字号、行高和响应式布局必须保证主要文本、按钮和错误提示可读，不得用低对比度颜色承载必要信息。
7. 路由切换或动态内容替换后应当更新页面标题、焦点位置或状态提示，让辅助技术用户知道上下文已变化。
8. 不得移除浏览器默认焦点样式，除非提供同等可见且一致的替代样式。

## 示例

正例：

```vue
<button type="button" :aria-label="t('dialog.close')" @click="closeDialog">
  <CloseIcon aria-hidden="true" />
</button>

<label for="mobile">{{ t("profile.mobile.label") }}</label>
<input id="mobile" v-model="mobile" aria-describedby="mobile-error" />
<p v-if="mobileError" id="mobile-error" role="alert">{{ mobileError }}</p>
```

反例：

```vue
<div class="button" @click="submit">提交</div>
<span class="error-dot"></span>
```

## 检查清单

- 交互控件是否使用语义元素或补足必要 ARIA 名称。
- 键盘是否能到达、操作并离开所有核心控件。
- 焦点样式、弹窗焦点管理和路由切换后的上下文是否可验证。
- 表单错误、加载和结果状态是否有文本表达。
- 信息是否避免只依赖颜色、图标、位置或动画。
- 第三方组件封装后是否满足项目可访问性约束。
