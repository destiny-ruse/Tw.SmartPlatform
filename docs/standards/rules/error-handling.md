# 错误处理规范

## 目标

错误处理规范用于降低异常被吞掉、错误语义不稳定、敏感信息泄露和恢复策略不可验证的风险。服务端必须把内部异常分类映射为稳定契约，并保留足够的诊断信息用于日志、追踪和告警。这样调用方、运维和测试可以根据同一错误模型判断可重试性和影响范围。

## 适用范围

本规范适用于 `apps/`、`services/`、`src/`、`backend/` 中 HTTP API、gRPC 服务、消息消费者、批处理任务、后台作业、公共框架中间件和集成测试。它覆盖异常分类、错误码映射、日志记录、重试决策、补偿处理和错误响应生成。它不适用于 UI 文案本地化、仅用于开发机调试的临时断言或第三方库内部未暴露异常。

## 规则

1. 服务必须区分输入校验错误、认证授权错误、领域规则错误、资源不存在、并发冲突、下游依赖错误和系统未知错误。
2. 面向调用方的错误必须映射为稳定错误码、协议状态和安全消息；不得直接返回异常类名、堆栈、SQL 或第三方原始错误。
3. 未知异常必须在边界层统一捕获、记录并转换为通用系统错误；不得让框架默认错误页或未序列化异常泄露给客户端。
4. 可重试错误和不可重试错误必须明确分类；不得对输入校验、权限失败或契约不兼容错误执行自动重试。
5. 日志必须记录错误码、异常分类、traceId、关键业务标识和安全脱敏后的上下文；不得记录密钥、令牌、完整证件号或原始敏感载荷。
6. 错误处理不得吞掉异常后继续返回成功；补偿、降级或忽略错误必须有显式业务语义和可观测记录。
7. 跨服务调用必须保留上游错误语义，并在本服务边界映射为本服务公开契约；不得把下游内部错误码无筛选透传给外部客户端。
8. 仓库自有代码显式抛出的异常必须提供简体中文消息；参数名、错误码、协议字段、算法名和第三方原文可以保留英文，不得依赖运行时默认英文异常消息作为稳定语义。

## 示例

正例：

```yaml
errors:
  ValidationError:
    httpStatus: 400
    code: VALIDATION_FAILED
    retryable: false
  DependencyTimeout:
    httpStatus: 503
    code: DEPENDENCY_TIMEOUT
    retryable: true
```

反例：

```java
catch (Exception ex) {
  return Response.ok(Map.of("success", true)).build();
}
```

## 检查清单

- 异常是否按输入、权限、领域、冲突、依赖和未知错误分类。
- 每类错误是否映射到稳定错误码、协议状态和安全消息。
- 未知异常是否在边界统一捕获，并避免框架默认错误泄露。
- 可重试性是否明确，且不可重试错误不会进入自动重试。
- 日志是否包含 traceId 和诊断上下文，同时完成敏感信息脱敏。
- 显式抛出的异常是否包含简体中文消息，且未依赖运行时默认异常消息。
