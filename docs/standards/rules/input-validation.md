# 输入校验规范

## 目标

输入校验规范用于降低未可信输入直接进入业务、存储或外部调用后造成安全漏洞、数据污染和错误响应不稳定的风险。系统必须在信任边界入口对请求、文件、消息和配置进行可审查校验，并将失败映射为稳定错误响应。这样后端、前端和 QA 可以明确哪些输入被接受、拒绝或脱敏。

## 适用范围

本规范适用于 `apps/`、`services/`、`src/`、`backend/`、`frontend/`、`mobile/` 中 HTTP 请求、gRPC 请求、异步消息、外部回调、批量导入、文件上传、环境配置、用户输入表单和测试数据。它覆盖路径参数、查询参数、请求头、请求体、文件名、文件类型、文件大小、编码、PII 和密钥输入。它不适用于完全由本进程生成且不跨信任边界的内部对象。

## 规则

1. 所有外部输入必须在信任边界入口校验类型、格式、长度、范围、必填性和枚举值；不得只依赖前端校验、数据库约束或下游服务报错。
2. 校验规则必须与公开契约一致，并在 OpenAPI、AsyncAPI、Proto 或配置 Schema 中表达；不得让实现接受契约禁止的宽松格式。
3. 文件上传必须校验文件大小、扩展名、MIME 类型、内容签名、文件名安全性和存储路径；不得只检查扩展名。
4. 对 URL、路径、SQL、命令、模板、正则和序列化输入必须使用安全解析或参数化 API；不得通过字符串拼接构造敏感操作。
5. 包含 PII、密钥、令牌或凭据的输入必须最小化接收、及时脱敏或加密，并避免写入普通日志和错误响应。
6. 校验失败必须返回稳定错误码和字段级明细；不得把内部校验异常或框架默认消息直接暴露给客户端。
7. 参数守卫和输入校验抛出的异常消息必须使用简体中文并可映射到字段级错误；不得把框架默认英文校验消息作为稳定契约。
8. 外部回调和异步消息必须先完成签名、来源、时间窗口和重放校验，再执行业务副作用。
9. 前端可以做即时体验校验，但后端必须重复校验所有安全和业务边界；不得把客户端视为可信执行环境。

## 示例

正例：

```yaml
upload:
  maxBytes: 5242880
  allowedMimeTypes: [image/png, image/jpeg]
  requireContentSignatureCheck: true
  sanitizeFileName: true
```

反例：

```python
sql = "select * from users where name = '" + request.query["name"] + "'"
```

## 检查清单

- 是否识别所有外部输入和信任边界，包括消息、回调、文件和配置。
- 类型、格式、长度、范围、必填性和枚举校验是否与契约一致。
- 文件上传是否校验大小、类型、内容签名、文件名和存储路径。
- PII、密钥和令牌是否避免进入日志、错误响应和不必要存储。
- 校验失败是否返回稳定错误码和字段级明细。
- 参数守卫和校验异常消息是否使用简体中文，且未依赖框架默认英文消息。
