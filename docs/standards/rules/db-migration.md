# 数据库迁移规范

## 目标

数据库迁移规范用于降低 schema 变更导致不可回滚发布、数据丢失和线上阻塞的风险。它要求结构变更、数据回填、兼容窗口和破坏性操作都具备可审查的执行与恢复路径。

## 适用范围

适用于关系型数据库和由迁移工具管理的 schema 变更、索引变更、约束变更、数据回填、数据修复脚本和与版本发布绑定的数据访问层调整。一次性本地开发数据库重建、测试夹具数据和不进入共享环境的临时实验不属于本规范范围，但不得影响受控迁移历史。

## 规则

1. 每个迁移必须有唯一、可排序、可追溯的名称，并说明业务意图；不得修改已经进入共享环境的历史迁移文件。
2. schema 变更必须优先采用向前兼容步骤，例如先加可空列或新表，再发布读写兼容代码，最后清理旧结构。
3. 每个迁移必须提供回滚策略；无法完全自动回滚时，必须说明数据保全、人工恢复步骤和停止条件。
4. 破坏性变更如删除表、删除列、收紧约束、批量重写数据或重建大索引，必须在评审中显式标注影响范围和回退方案。
5. 数据回填必须可分批、可重入、可观测，并限制单批影响范围；不得在单个事务中无界扫描或锁定大表。
6. 迁移脚本必须维护审计字段、租户边界和软删除语义，不得绕过应用层约束造成跨租户写入。
7. 新增约束或唯一索引前必须验证既有数据质量，并在迁移说明中记录清理或例外策略。
8. 应用代码与迁移的发布顺序必须明确，涉及回滚时不得使旧版本代码无法读取当前 schema。

## 示例

正例：

```sql
-- 0018_add_order_status.sql
alter table sales_order add column status varchar(32) null;
update sales_order
set status = 'created'
where status is null
  and id between :start_id and :end_id;
```

正例说明：先兼容新增字段，再通过可分批、可重入条件回填数据。

反例：

```sql
alter table sales_order drop column legacy_status;
```

反例说明：直接删除旧列缺少兼容窗口、影响范围和回滚策略。

## 检查清单

- 迁移名称是否唯一、可排序并说明业务意图。
- 是否存在向前兼容发布顺序和旧版本代码读取边界。
- 回滚策略是否覆盖 schema、数据和应用版本。
- 破坏性变更是否显式标注影响范围、验证方式和恢复步骤。
- 数据回填是否可分批、可重入、可观测且维护租户和审计字段。
- 新约束或索引是否已验证历史数据质量。
