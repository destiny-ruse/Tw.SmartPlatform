# 个人信息处理规范

## 目标

个人信息处理规范用于降低个人信息被过度收集、错误存储、未授权访问、日志泄露或测试数据复用造成的合规与安全风险。系统必须在识别、收集、传输、存储、展示、日志和测试环节保持一致的最小化与脱敏要求。这样提交者和评审者可以直接判断变更是否扩大了个人信息处理范围。

## 适用范围

本规范适用于 `apps/`、`services/`、`src/`、`backend/`、`frontend/`、`mobile/`、数据库迁移、数据修复脚本、API 契约、导入导出文件、报表、日志、监控事件、测试夹具和演示数据中涉及个人信息的处理。个人信息包括可识别自然人的姓名、手机号、邮箱、证件号、地址、账号标识、设备标识、精确位置、认证主体 ID 以及与这些字段组合后可识别个人的数据。本规范不适用于完全匿名化且无法重新识别的聚合统计，但脱敏、哈希或截断后仍可关联个人的数据仍在本规范范围内。

## 规则

1. 新增字段、API、事件、报表或导入导出能力时必须识别是否包含 PII，并在契约、字段说明或评审记录中标明处理目的。
2. 系统必须按目的最小化收集、传输和保存 PII；不得为了未来可能使用而新增非必要个人信息字段。
3. 展示、日志、错误响应、监控标签和导出文件必须按场景脱敏或屏蔽 PII；不得输出完整证件号、手机号、邮箱、地址、令牌或认证标识。
4. PII 的读取、修改、导出和批量查询必须受访问控制保护，并应当使用稳定主体 ID 记录审计信息；不得依赖前端隐藏字段实现权限边界。
5. API、消息和数据库中保存 PII 时必须使用明确字段名和类型，并避免把多种个人信息拼接到不可解析的自由文本字段。
6. 测试、演示、截图和文档示例必须使用合成数据或已脱敏数据；不得复制生产个人信息到本地、非生产环境或仓库。
7. 数据修复、排障和临时导出可以在必要范围内处理 PII，但必须限定字段、时间范围和接收方，并在完成后删除临时文件。
8. 发现 PII 泄露到日志、仓库、错误响应、制品或非授权环境时，必须停止继续扩散、移除可访问副本并按事件边界处理；不得在同一提交中掩盖泄露痕迹或继续复用泄露数据。

## 示例

正例：

```json
{
  "customerId": "usr_9f2a1c",
  "phoneMasked": "138****1234",
  "emailMasked": "li***@example.com"
}
```

正例说明：接口返回稳定主体 ID 和脱敏展示字段，避免暴露完整联系方式。

反例：

```sql
insert into test_users (name, phone, id_card)
values ('张三', '13800138000', '110101199001011234');
```

反例说明：测试数据不得使用真实姓名、手机号和证件号组合。

## 检查清单

- 新增或变更的数据字段、API、事件、报表和导出文件是否已识别 PII。
- 收集、传输和保存的 PII 是否与当前业务目的直接相关并保持最小化。
- 日志、错误响应、监控、页面展示和导出文件是否已脱敏或屏蔽敏感字段。
- PII 读取、修改、导出和批量查询是否有服务端访问控制和审计字段。
- 测试夹具、演示数据、截图和文档示例是否使用合成或脱敏数据。
- 泄露响应是否限定在停止扩散、移除副本、轮换相关凭据和补充验证边界内。
