# 缓存约定规范

## 目标

缓存约定规范用于降低缓存键冲突、过期策略缺失和脏数据扩散带来的运行风险。它要求缓存设计在命名、TTL、失效和降级边界上可审查，让不同服务和语言栈对缓存一致性形成相同预期。

## 适用范围

适用于后端服务、批处理任务、消息消费者和公共库中使用的本地缓存、分布式缓存、查询结果缓存和幂等缓存条目。也适用于缓存配置、缓存键生成器、失效事件、回源逻辑和缓存相关代码评审；不适用于数据库主键设计、浏览器 HTTP 缓存策略或仅存在于单次函数调用内的临时变量。

## 规则

1. 缓存键必须包含稳定命名空间、业务域、实体或查询标识和版本片段，不得使用裸 ID、拼写不明的缩写或跨服务共享的隐式键。
2. 每类缓存必须声明 TTL 选择依据；强一致读取不得使用未说明一致性边界的长 TTL。
3. 缓存值可以冗余派生数据，但必须说明来源数据、序列化格式和兼容版本，不得把缓存作为唯一事实来源。
4. 写入或状态变更路径必须定义失效策略，可以使用删除、版本递增、事件驱动失效或短 TTL，但不得只依赖人工清理。
5. 热点 key、批量回源和缓存未命中路径应当具备互斥加载、请求合并、限流或抖动 TTL 等防击穿措施。
6. 缓存穿透风险高的查询必须限制空值缓存的 TTL 和适用条件，不得永久缓存缺失结果。
7. 允许读取陈旧数据的场景必须标明最大可接受陈旧时间、用户可见影响和恢复方式；涉及权限、余额、库存等高风险数据时不得默认使用陈旧缓存。
8. 缓存指标和日志应当能区分命中、未命中、回源失败、失效和序列化错误，便于定位容量或一致性问题。

## 示例

正例：

```typescript
const key = `smart-platform:catalog:product:v2:${tenantId}:${productId}`;
await cache.set(key, productView, { ttlSeconds: 300, jitterSeconds: 30 });
```

正例说明：键包含系统命名空间、业务域、实体版本、租户和实体 ID，TTL 有明确上限并带有抖动。

反例：

```typescript
await cache.set(productId, productView);
```

反例说明：裸 ID 容易与其他缓存冲突，且没有 TTL、版本和租户边界。

## 检查清单

- 缓存键是否包含命名空间、业务域、版本和必要的租户或主体边界。
- 每类缓存是否声明 TTL、TTL 抖动和陈旧数据可接受边界。
- 写入、删除、权限变更或数据修复后是否有失效或版本切换策略。
- 回源路径是否有防击穿、防穿透或限流措施。
- 缓存值结构变更是否考虑序列化兼容和旧 key 清理。
- 指标或日志是否能支持命中率、回源失败和失效问题排查。
