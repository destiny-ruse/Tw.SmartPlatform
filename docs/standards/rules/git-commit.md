# Git 提交规范

## 目标

Git 提交规范用于降低历史不可审查、回滚粒度过大、生成产物来源不清和本地文件误入仓库的风险。提交记录必须表达清晰意图，让审阅者、发布人员和自动化工具可以判断变更类型、影响范围和恢复路径。规范还确保 `.gitignore`、生成文件和提交说明共同维护仓库边界。

## 适用范围

本规范适用于所有进入仓库历史的代码、测试、文档、配置、CI、规范、工具脚本和生成产物提交。它不适用于本地未提交改动、临时 stash、外部依赖仓库历史或由上游项目维护的第三方镜像提交。

## 规则

1. 提交信息必须使用 `<type>: <summary>` 格式，类型可以是 `feat`、`fix`、`docs`、`test`、`refactor`、`chore`、`build`、`ci`；不得使用空泛摘要如 `update`、`changes`、`fix bug`。
2. 摘要必须用祈使或结果导向短句说明变更意图，并保持可搜索；不得只写任务编号而不说明内容。
3. 单个提交应当表达一个可审查、可回滚的变更意图；不得混合无关功能、重构、格式化、依赖升级和生成产物。
4. 影响契约、规范、部署或生成产物的提交必须在正文或 PR 说明中写明验证命令、生成命令或影响范围。
5. 生成产物必须与源文件变更位于同一提交或相邻提交，并能通过提交说明或变更记录找到生成命令；不得提交无法复现的生成文件。
6. `.gitignore` 是提交边界的最高依据；已被忽略的文件或目录不得通过 `git add -f`、`git update-index` 等方式强行纳入提交。
7. 若被忽略的文件确实需要纳入版本管理，必须先修改忽略规则并说明原因，再按普通提交流程纳入历史。
8. 提交不得包含临时调试输出、个人环境文件、密钥、IDE 私有状态或未解释的大规模格式化噪音。
9. 修复 review 反馈时应当保持提交可审查；可以 squash 同一主题的细碎修正，不得用无意义的 `fix review` 掩盖实际修改。

## 示例

正例：

```shell
git commit -m "docs: expand general engineering standards rules"
git commit -m "ci: verify standards index generation"
```

反例：

```shell
git add -f .env
git commit -m "update"
```

## 检查清单

- 提交信息是否符合 `<type>: <summary>`，且摘要说明真实意图。
- 单个提交是否保持可审查、可回滚，没有混入无关变更。
- 生成产物是否与源文件和生成命令关联。
- 提交文件是否命中 `.gitignore`；若命中，是否先修改忽略规则并说明原因。
- 是否排除了本地环境、密钥、调试输出和无解释格式化噪音。
- 影响规范、部署、契约或 CI 的提交是否说明验证结果。
