# 规范撰写规范

## 目标

本规范定义工程规范文档的最小结构、章节组织、引用方式和变更治理要求。所有进入 `docs/standards/` 的正式规范资料都必须面向人员直接阅读。

## 适用范围

本规范适用于以下正式资料：

1. `rules/` 下的标准正文。
2. `processes/` 下的流程规范。
3. `decisions/` 下的长期有效工程决策记录。
4. `references/` 下的参考资料。

生成索引、运行缓存、临时查询结果、工具输出、front matter 元数据和机器可读锚点不属于正式规范资料，不应存放在 `docs/standards/` 正文中。

## 文档结构

规范文档必须使用 Markdown，并包含以下结构：

1. 一个一级标题。
2. 语义清晰的二级标题。
3. 可独立评审的规则、流程、决策或参考内容。
4. 必要的示例、检查清单或相关规范。
5. 变更记录。

文档正文不使用 YAML front matter，不写 `machine_rules`、`review_after`、`roles` 等机器字段，也不使用隐藏锚点注释。

## 状态管理

规范状态通过目录位置、评审结论和变更记录管理：

1. 草稿资料应先在设计文档、RFC 或评审分支中讨论。
2. 生效资料合入 `docs/standards/` 后可被开发、测试、运维和评审流程引用。
3. 废弃或替代关系必须在正文中明确说明，并保留必要历史。

## 引用要求

引用规范结论时，至少说明：

1. 文档标题。
2. 章节标题。
3. Markdown 文件路径。
4. 适用版本或变更记录中的日期。

## 变更流程

新增或重大修改规范时，应先创建 RFC。长期有效的工程决策应补充决策记录。小型修订可以直接修改规范，但必须更新变更记录。

修改规范后必须运行：

```powershell
python tools\tw-memory\tw_memory.py generate --format brief
python tools\tw-memory\tw_memory.py check --format brief
```

当前仓库没有独立的标准索引脚本；不得在规范正文中声明依赖不存在脚本的流程。
